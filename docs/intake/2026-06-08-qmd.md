# Intake — QMD (tobi/qmd) (2026-06-08)

- **Type** : MCP / retrieval tool (couche N1-2 rappel)
- **Source** : https://github.com/tobi/qmd
- **Récence / obsolescence** : actif 2026 / **medium** (mono-mainteneur, jeune)
- **Résumé** :
  - Recherche locale de docs : **BM25 + vector + LLM reranking**, avec **query-expansion** automatique (Qwen3 1.7B local).
  - Sert un **MCP server natif** (`query`, `get`, `multi_get`) → les agents lisent via tool, rien à écrire.
  - Stack TypeScript, **Node ≥22** ✓ (v22.16.0 confirmé). 100 % local, **pas de clé API**.
  - Télécharge **~2 GB de modèles** dans `~/.cache/qmd/models/` (EmbeddingGemma 300M, Qwen3 Reranker 0.6B, Query-Expansion 1.7B).

## Fit
- Surface : **memory / context-manager**. Couvre les niveaux **N1 (stockage) + N2 (rappel)** sur les registres Markdown `data/memory/` et `docs/knowledge/`.
- Pas un doublon : il **remplace** le plan custom `index.db` SQLite-FTS5 de `memory-patterns.md`, en plus puissant (le custom = BM25 seul).
- N'empiète **pas** sur N3 (jugement) : les 5 registres restent écrits par le Memory Keeper.

## Coûts
- **Install** : `npm`/`uv` + **2 GB de modèles** au premier run + câblage MCP dans le worker. Effort moyen, tokens ~0 (pas de LLM build-time).
- **Maintenance** : suivre le repo `tobi/qmd` (mono-mainteneur → risque d'abandon). Re-index auto des collections.
- **Retrait** : **facile SI** on l'isole derrière une interface `MemoryRetriever` (swap 1 fichier). Sans interface : moyen (les agents appelleraient `qmd.query` en dur). → l'interface est la condition de réversibilité.

## Score
project_fit **5** / token_efficiency **5** (retrieval on-demand, -92% vs injection naïve) / safety **4** (local, mais DL modèles = supply-chain mineure) / implementation_effort **3** (2 GB + MCP) / evidence_maturity **3** (jeune, mono-mainteneur) / user_value **4** / phase_compatibility **4** (Phase 4, mais pas critique au MVP)

## KILL check
**Aucun veto bloquant.** 100 % local + pas de clé API → passe §11. MCP = process localhost (pas de host externe, §5 ok). Réserve : télécharge des modèles depuis HF → **audit supply-chain léger** avant install (pin de version/hash).

## Décision
**backlog_next (Phase 4.x — fast-follow, pas le MVP)** — Adopté comme **retriever cible**, mais **après** que le MVP FTS5 ait prouvé les 5 registres + le pont. On le pose **derrière l'interface `MemoryRetriever`** pour que le swap FTS5→QMD soit un changement d'un fichier. Justif : le *gate* de sortie Phase 4 (retrouver "BDR", "Mem0 cloud"…) est satisfait par BM25/FTS5 ; les 2 GB + MCP + maturité jeune ne se justifient qu'à corpus plus gros. Voir **ADR 0003**.

## Appropriation (si retenu)
- Version MAS : `MemoryRetriever` interface dans `packages/memory/` ; impl `FtsRetriever` (MVP) puis `QmdRetriever` (upgrade).
- Collections MCP : `mas-knowledge` = [`docs/knowledge/`, `docs/workflows/`] ; `mas-memory` = [`data/memory/`].
- Moins cher : retrieval **on-demand via tool**, jamais d'injection auto au démarrage (anti-pattern memory-patterns §297).

## Plan (si retenu)
- Phase : **4.x** (après le MVP FTS5 du gate Phase 4).
- Fichiers : `packages/memory/src/retriever.ts` (interface + impls), config MCP du worker.
- Mini-gate QMD : modèles téléchargés + `qmd.query` répond + audit supply-chain + l'interface reste stable.
- Ne PAS faire : brancher QMD **avant** d'avoir les registres + le pont FTS5 verts (sinon on bloque Phase 4 sur une dépendance lourde).

## Ré-audit
Quand le corpus mémoire d'un projet **> 200 items**, OU dès qu'un besoin de **rappel sémantique/flou** apparaît (requêtes non exactes). Re-check repo si > 6 mois sans commit.
