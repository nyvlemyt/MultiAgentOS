# Intake — RooFlow (2026-07-10)

- **Type** : repo (système "Memory Bank" + modes custom pour l'extension VS Code Roo Code)
- **Source** : https://github.com/GreatScottyMac/rooflow
- **Récence / obsolescence** : dernière release v0.5.6 (4 mai 2025, soit ~14 mois au moment de l'audit) / **high** — projet marqué expérimental, lié à l'écosystème Roo Code (pas Claude Code), 31 issues ouvertes sans release depuis.
- **Résumé** :
  - Memory Bank = 5 fichiers Markdown persistants (`activeContext.md`, `decisionLog.md`, `productContext.md`, `progress.md`, `systemPatterns.md`) maintenus automatiquement par l'agent.
  - 5 modes custom (Flow-Code / Architect / Ask / Debug / Orchestrator) avec niveaux d'accès mémoire (Ask = read-only).
  - Commande `UMB` ("Update Memory Bank") pour forcer la synchro session → mémoire.
  - System prompts en YAML (claim non sourcé de réduction de tokens vs Markdown).
  - Install via scripts shell/batch + Python 3 + PyYAML ; Apache 2.0 ; pas de clé API ni service payant.

## Guardrails (étape 0)

- **Subscription-only (§11)** ✓ — pas de PAYG, pas de clé API. *(pas de kill ici)*
- **Memory Keeper seul écrivain (§8)** ✗ — le cœur de RooFlow est l'écriture mémoire **automatique et directe** par chaque mode ("updated automatically based on significant events within each mode") = free agent writes. Même conflit que le pattern Karpathy (dossier 2026-06-27), déjà tranché par ADR 0004 : tout passe par `captureCandidates()` → promotion Keeper.
- **Local-first** ✓ — fichiers Markdown locaux.
- **Pas de framework sans ADR** — RooFlow est un framework de prompts + scripts d'install lié à un hôte tiers (Roo Code, pas Claude Code) : l'adopter tel quel exigerait une ré-adaptation complète.

## KILL check

- PAYG / clé API → non. *(pas de veto §11)*
- Free agent writes sur la mémoire → **veto §8** sur l'implémentation telle quelle.
- Framework lié à un hôte tiers (Roo Code) + stale 14 mois → dup-no-better.

## Décision

**`reject`** (T0 — dup-no-better) — Les 5 fichiers du Memory Bank mappent 1:1 sur les 5 registres déjà en doctrine (`project-doctrine.md`, ADR 0003/0004), souvent en version supérieure (SQLite pour le progress, QMD pour le retrieval, Keeper-gating pour l'écriture). Le mécanisme central (écriture mémoire directe par chaque mode) viole §8. Repo stale (mai 2025), expérimental, couplé à Roo Code. Rien à distiller que le dossier Karpathy (2026-06-27) n'ait déjà tranché.

## Ré-audit

Condition : uniquement si RooFlow reprend une activité significative (nouvelle release majeure) **et** introduit un mécanisme absent de MAOS (ex. un retrieval ou une consolidation nouvelle). Pas de date fixe — le reject est structurel (doublon), pas conjoncturel.
